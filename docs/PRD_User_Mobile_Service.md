# MIND BREEZE — 고객용 모바일 서비스 PRD

버전 1.0 | 2026.03  
기준 아트팩트: `design/front-master.pen` (관리자용 웹 구조 참조), `docs/MIND_BREEZE_Service_Plan.md`

---

## 1) 문서 목적

현재 구축된 관리자용 웹 서비스의 정보구조(IA)와 운영 흐름을 기준으로, 고객(내담자/참여자) 대상 모바일 우선 서비스를 최소 범위로 정의한다.  
본 PRD의 목표는 **빠른 출시 가능한 MVP**를 만들되, 추후 고도화(네이티브 확장, BLE 분석 심화) 가능한 구조를 확보하는 것이다.

---

## 2) 디자인 참고 요약 (관리자 웹 현황)

`front-master.pen`에서 확인한 관리자 핵심 화면군:

- `App V2 — Home Dashboard`
- `Schedule (Daily/Weekly/Monthly)`
- `Chat (Main)`
- `Care - Main`
- `Client Management`
- `Notification (Main/Template)`
- `Board (Main)`
- `Settings` 및 하위(`Notification`, `Schedule`, `LINK BAND`, `Security`, `Help`, `Theme`)
- `Journal by Client/Date`

즉, 관리자 서비스는 이미 **일정·채팅·케어·설정·게시판·알림** 중심으로 구조화되어 있으며, 고객용도 동일 도메인을 단순화해 대응하는 것이 정합성이 높다.

---

## 3) 제품 목표 (고객용 MVP)

### 3.1 목표

1. 고객이 상담/수업 참여에 필요한 핵심 기능을 모바일에서 빠르게 수행
2. 상담사와의 소통/일정 확인/케어 수행을 하나의 앱 경험으로 통합
3. 웹앱 중심으로 개발하고, 디바이스 의존 기능(Push, LINK BAND BLE)만 선택적으로 보강

### 3.2 비목표 (이번 MVP 제외)

- 커뮤니티 고도화 기능(랭킹, 포인트, 대형 피드 알고리즘)
- 고급 분석 대시보드(장기 통계, 실험 기능)
- 완전 네이티브 UI 재구현(React Native full rewrite)

---

## 4) 대상 사용자

- **내담자(1:1 상담 고객)**: 예약 확인, 채팅 상담, 케어 과제 수행, 상담 기록 확인
- **명상 참여자(그룹 수업 고객)**: 일정 확인, 공지/게시판 확인, 기본 채팅, 케어 콘텐츠 수행

---

## 5) 핵심 기능 범위 (MVP)

하단 탭 5개를 기준으로 모바일 IA를 구성한다.

1. **일정**
2. **채팅**
3. **케어**
4. **게시판**
5. **설정**

### 5.1 일정

- 내 일정 목록(예정/완료/취소)
- 일정 상세(일시, 상담사/수업명, 참여 링크/장소, 준비사항)
- 리마인더 수신 설정
- 캘린더 뷰(월/주 중 1개 우선, 목록 뷰 필수)

### 5.2 채팅

- 상담사 1:1 채팅방
- 시스템 안내 메시지(예약 확정/변경/리포트 도착)
- 파일 첨부(이미지) 선택 지원 여부는 P1
- 읽음 상태 및 최근 메시지 기준 정렬

### 5.3 케어

- 오늘의 케어 과제(명상/호흡/저널링)
- 수행 체크(완료, 소요시간, 간단 메모)
- 상담사 피드백 확인
- 기록 히스토리(날짜별 목록)

### 5.4 게시판

- 공지/콘텐츠 목록
- 게시글 상세 열람
- 좋아요/댓글은 P1 (MVP는 열람 중심)

### 5.5 설정

- 프로필/계정
- 알림 설정(Push, 채팅 알림, 일정 리마인더)
- LINK BAND 연결/해제 및 연결 상태
- 보안/도움말/약관

## 6) 기술 전략 (중요 의사결정)

### 6.1 기본 전략

- **주 서비스:** React 기반 모바일 웹앱(반응형)
- **보강 기능:** Push Notification + LINK BAND BLE

### 6.2 권장 구현안

1. 1차 릴리스: 웹앱 중심으로 핵심 기능 출시
2. 2차 릴리스: 디바이스 기능 안정화를 위해 앱 셸 도입
   - 후보: React Native 최소 셸 또는 Capacitor 셸
   - 역할: Push 토큰/수신, BLE 연결, 권한 제어
3. UI/도메인 로직은 웹 기준 유지, 네이티브는 브리지 기능만 담당

### 6.3 이유

- Web Bluetooth/iOS 제약으로 BLE를 웹만으로 보장하기 어려움
- 웹 푸시는 브라우저/OS 정책 차이가 커 전달 안정성 이슈 가능
- 따라서 제품 본체는 웹, 디바이스 기능은 네이티브 보강이 비용 대비 효율적

## 7) 기능 요구사항 (FR)

### 7.1 공통/인증

- FR-AUTH-01: 휴대폰/이메일 기반 로그인
- FR-AUTH-02: 고객 권한에서 접근 가능한 메뉴만 노출

### 7.2 일정

- FR-SCH-01: 일정 목록 조회(상태 필터 포함)
- FR-SCH-02: 일정 상세 조회
- FR-SCH-03: 리마인더 알림 수신

### 7.3 채팅

- FR-CHAT-01: 1:1 채팅방 목록/입장
- FR-CHAT-02: 메시지 송수신(텍스트)
- FR-CHAT-03: 시스템 알림 메시지 자동 생성(예약/리포트 이벤트)

### 7.4 케어

- FR-CARE-01: 케어 과제 목록/상세 조회
- FR-CARE-02: 케어 수행 결과 기록(완료/메모/시간)
- FR-CARE-03: 과거 수행 기록 조회

### 7.5 게시판

- FR-BRD-01: 게시글 목록/상세 조회
- FR-BRD-02: 카테고리(공지/가이드) 필터

### 7.6 설정/디바이스

- FR-SET-01: 알림 ON/OFF 설정
- FR-SET-02: LINK BAND 연결/재시도/해제
- FR-SET-03: BLE 미지원 환경 안내 및 대체 UX 제공

---

## 8) 비기능 요구사항 (NFR)

- NFR-01 보안: TLS 전송, 토큰 기반 인증, 민감정보 접근 제어
- NFR-02 개인정보: 케어/채팅 기록 보관 정책과 동의 절차
- NFR-03 성능: 핵심 화면 초기 로드 최적화, 채팅 지연 최소화
- NFR-04 호환성: iOS/Android 모바일 브라우저 우선 지원 범위 명시
- NFR-05 안정성: Push 실패 시 인앱 배지/목록 폴백 제공

## 9) 사용자 플로우

1. 로그인
2. 홈(또는 일정 탭)에서 오늘 일정 확인
3. 일정 시작 시 채팅/세션 진입
4. 세션 후 케어 과제 확인 및 수행
5. 게시판에서 공지/가이드 확인
6. 설정에서 알림·LINK BAND 상태 관리

## 10) MVP 우선순위

### P0 (필수)

- 일정 목록/상세
- 채팅 기본 송수신
- 케어 과제 조회/완료
- 게시판 읽기
- 설정(알림, LINK BAND 연결 상태)

### P1 (차기)

- 채팅 첨부/검색
- 게시판 상호작용(댓글/좋아요)
- 케어 통계 시각화

---

## 11) 성공 지표 (초기)

- 주간 활성 사용자(WAU)
- 일정 확인 후 세션 참여율
- 채팅 응답률(24시간 내)
- 케어 과제 완료율
- 푸시 수신 성공률, LINK BAND 연결 성공률

---

## 12) 오픈 이슈

1. 채팅을 자체 구축할지, 외부 채널(예: 카카오)과 병행할지 최종 결정
2. iOS에서 BLE 연결 보장 수준(웹 한정 vs 앱 셸 필수)
3. Push 채널 정책(웹 푸시 단독 vs 네이티브 푸시 병행)
4. 게시판 운영 권한(읽기 전용 vs 고객 작성 허용)

---

## 13) 개발 메모 (권장)

- 코드 구조는 `web-core`(공통 도메인) + `device-bridge`(push/ble 어댑터)로 분리
- 관리자 웹의 도메인 모델(일정/채팅/케어/설정)을 재사용 가능한 API 계약으로 정리
- 고객용은 화면 단순화, 용어는 비전문가 친화적으로 재작성
